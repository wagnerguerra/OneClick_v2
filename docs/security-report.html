CriticoSEC-01 - Segredos, certificados e dumps rastreados no Git

Foram encontrados arquivos sensiveis versionados: certs/certificado.pfx, certs/key.pem, certs/cert.pem, database-2026-04-11T19-17-47.sql, database-dump-2026-04-13.sql e apps/android-parallel/credentials.json.

Evidencias: git ls-files certs database-*.sql apps\android-parallel\credentials.json; .gitignore:57 declara que certs/ e versionada; certs/ contem PFX e chaves privadas.

Correcao recomendada:
  • Revogar/rotacionar imediatamente certificados, chaves, tokens e credenciais possivelmente expostos.
  • Remover esses arquivos do historico Git com ferramenta propria para rewrite de historico e invalidar clones antigos.
  • Adicionar certs/, *.pfx, *.pem, credentials.json e dumps SQL ao .gitignore.
  • Substituir por referencias a secret manager ou variaveis de ambiente; manter exemplos sem dados reais.

CriticoSEC-02 - Webhook WhatsApp pode aceitar requisicoes sem validacao de assinatura

O controller valida X-Hub-Signature-256 somente se req.rawBody existir. Porem o bootstrap preserva rawBody apenas para /api/stripe/webhook. Resultado: no fluxo atual, rawBody tende a ficar ausente no WhatsApp e a validacao nao bloqueia o POST.

Evidencias: apps/api/src/main.ts:28-30 preserva raw body apenas para Stripe; apps/api/src/whatsapp/whatsapp-webhook.controller.ts:31-35 usa if (raw && !verificarAssinatura(...)).

Correcao recomendada:
  • Preservar rawBody para /api/whatsapp/webhook no body parser.
  • Falhar fechado: se WHATSAPP_APP_SECRET estiver configurado, exigir assinatura e raw body; retornar 401 quando faltar.
  • Adicionar teste automatizado cobrindo POST sem assinatura, assinatura invalida e assinatura valida.

CriticoSEC-03 - Uploads publicos permitem sobrescrever certificados PFX globais

O controller /api/upload nao mostra autenticacao/guard local. As rotas POST /api/upload/certificado e POST /api/upload/certificado-pf gravam nomes fixos (certificado.pfx e certificado-pf.pfx), possibilitando substituicao de certificado se expostas.

Evidencias: apps/api/src/upload/upload.controller.ts:59-79, apps/api/src/upload/upload.controller.ts:86-106; arquivos sao servidos por sendFile em apps/api/src/upload/upload.controller.ts:113-143.

Correcao recomendada:
  • Exigir sessao e permissao administrativa nas rotas de upload/consulta de arquivos.
  • Nao usar nomes globais fixos para PFX; vincular arquivo a tenant/empresa, owner e registro no banco.
  • Armazenar PFX cifrado em repouso, fora de pasta publica, e nunca servir PFX por rota estatica.
  • Validar MIME por assinatura do arquivo, extensao, tamanho e conteudo esperado.

CriticoSEC-04 - Download de backup sem autenticacao aparente

A rota direta /api/backup/download/:filename e registrada manualmente no Express, fora do tRPC e sem verificacao de sessao. Mesmo com bloqueio simples de .. e extensao .zip, um nome previsivel de backup pode causar exfiltracao completa.

Evidencia: apps/api/src/main.ts:66-80.

Correcao recomendada:
  • Remover a rota do bootstrap e mover para controller protegido por master/global admin.
  • Gerar URLs assinadas de curta duracao para downloads de backup.
  • Registrar auditoria com usuario, IP, timestamp, arquivo e resultado.

AltoSEC-05 - CORS e trusted origins aceitam rede privada de forma ampla

A API aceita origens LAN RFC1918 e localhost com credenciais. O Better Auth tambem adiciona dinamicamente origens contendo 192.168., 10. ou 172.. Em ambientes corporativos, qualquer host interno comprometido pode tentar operar contra a API usando cookies/sessoes do usuario.

Evidencias: apps/api/src/main.ts:36-51; apps/api/src/auth/auth.service.ts:60-75; API escuta em 0.0.0.0 em apps/api/src/main.ts:82.

Correcao recomendada: em producao, usar allowlist explicita por dominio HTTPS; separar politica de dev e prod; adicionar testes de CORS; garantir cookies Secure, HttpOnly e SameSite adequado.

AltoSEC-06 - Banco e Redis expostos no host com senha padrao

O compose publica Postgres em 5435 e Redis em 6380. O Postgres usa POSTGRES_PASSWORD=postgres; Redis nao mostra senha. Isso e aceitavel apenas para dev local isolado, mas perigoso se reutilizado em servidor ou maquina acessivel na LAN.

Evidencias: docker-compose.yml:6-12 e docker-compose.yml:21-23.

Correcao recomendada: usar senhas fortes via secret/env, bind em 127.0.0.1 quando local, nao publicar portas em producao, habilitar autenticacao no Redis e restringir rede Docker/firewall.

AltoSEC-07 - Console SQL master-only executa SQL arbitrario em producao

O sqlConsole e restrito a master, mas executa qualquer SQL via $queryRawUnsafe/$executeRawUnsafe. Um sequestro de conta master vira controle total do banco, inclusive alteracao/destruicao de dados.

Evidencias: apps/api/src/sql-console/sql-console.router.ts:8-13; apps/api/src/sql-console/sql-console.service.ts:83-98.

Correcao recomendada: exigir MFA e reautenticacao recente; limitar por ambiente; usar role DB read-only por padrao; bloquear comandos destrutivos sem aprovacao dupla; registrar auditoria imutavel da query e resultado resumido.

AltoSEC-08 - Limites de payload altos sem throttling global consistente

A API aceita JSON e urlencoded de ate 100 MB globalmente. Ha upload de ferramentas com 300 MB. Alguns endpoints publicos tem rate limit manual, mas nao ha evidencia de throttling global no Nest. Isso amplia risco de DoS por memoria/CPU e upload abusivo.

Evidencias: apps/api/src/main.ts:25-34; apps/api/src/ferramentas/ferramentas.controller.ts usa limite de 300 MB; apps/api/src/lead/lead.controller.ts implementa rate limit in-memory pontual.

Correcao recomendada: aplicar rate limiting central por IP/usuario/rota, quotas por tenant, streaming para uploads grandes, limites menores por endpoint e protecao no proxy reverso.

AltoSEC-09 - Dependencias vulneraveis no lockfile

pnpm audit encontrou 50 vulnerabilidades: 1 critica, 18 altas, 26 moderadas e 5 baixas. A critica envolve vitest <3.2.6. Altas incluem xlsx 0.18.5 com Prototype Pollution/ReDoS e dependencias transitivas como tar/undici.

Evidencia: pnpm audit --audit-level critical reportou GHSA-5xrq-8626-4rwp em apps__web>vitest; pnpm audit --json reportou xlsx, tar, undici, js-yaml e outros.

Correcao recomendada: atualizar vitest, substituir ou isolar xlsx, atualizar arvore Expo/tar/undici, e colocar audit no CI com politica por severidade e contexto runtime/dev.

AltoSEC-10 - Electron launcher expoe IPCs de alto privilegio

O Electron usa contextIsolation e desativa nodeIntegration, o que e positivo. Mesmo assim, o preload expoe funcoes para deploy, SSH/Git, iniciar/parar servicos, Docker, abrir URLs externas e iniciar processos. Qualquer XSS no index.html vira capacidade operacional local.

Evidencias: scripts/launcher/preload.js:3-101; scripts/launcher/main.js:2363, scripts/launcher/main.js:2879, scripts/launcher/main.js:2917-2954, scripts/launcher/main.js:3275-3278.

Correcao recomendada: validar tipos e allowlists no main process, restringir openExternal a http/https e dominios confiaveis, exigir confirmacao para deploy/execucao, adicionar CSP no HTML local e reduzir a API do preload ao minimo necessario.

AltoSEC-11 - Build Next ignora erros TypeScript e ESLint

O build de producao esta configurado para ignorar erros de TypeScript e ESLint. Isso reduz a chance de bloquear regressao de seguranca, quebra de tipos em autorizacao e imports incorretos.

Evidencias: apps/web/next.config.ts:12-13.

Correcao recomendada: remover os ignores, separar baseline legado, falhar CI para novos erros e manter excecoes temporarias com prazo.

MedioSEC-12 - Download tokens do webapp dependem de segredo minimo de 16 caracteres

O webapp usa JWT HS256 de 15 minutos para downloads, mas o schema aceita JWT_SECRET com minimo de 16 caracteres. Para tokens assinados, 32 bytes aleatorios ou mais e recomendavel.

Evidencias: webapp/webapp-01/apps/api/src/env.ts:31; webapp/webapp-01/apps/api/src/tokens.ts:14-24.

Correcao recomendada: exigir segredo >=32 bytes aleatorios, rotacao controlada e separar segredo por ambiente.

MedioSEC-13 - Rate limit do webapp libera polling de jobs sem limite

O Fastify webapp tem rate limit, mas isenta qualquer GET contendo /jobs/. Isso reduz falsos positivos de polling, mas pode permitir abuso de status/download probing se combinado com outros vetores.

Evidencias: webapp/webapp-01/apps/api/src/server.ts:223-227.

Correcao recomendada: aplicar limite mais alto, mas nao ilimitado, por IP/job/tenant; diferenciar status de download; usar cache curto para status.

MedioSEC-14 - Arquivos enviados sao servidos sem politica de tipo/conteudo

/api/upload/:filename serve qualquer arquivo permitido pelo filtro, bloqueando apenas executaveis por extensao. Imagens, HTML, SVG ou PDFs maliciosos podem causar XSS/reflected content ou ataques no navegador se abertos no mesmo dominio autenticado.

Evidencias: apps/api/src/upload/upload.controller.ts:23-46 e apps/api/src/upload/upload.controller.ts:113-124.

Correcao recomendada: servir anexos em dominio cookieless ou S3 com headers seguros; forcar Content-Disposition: attachment para tipos nao confiaveis; validar MIME real e bloquear HTML/SVG ativos.

MedioSEC-15 - Bearer auth habilitado amplia superficie de token

O Better Auth habilita plugin bearer() para mobile. Isso e funcional, mas exige controles de armazenamento, expiracao, revogacao e monitoramento equivalentes aos cookies.

Evidencia: apps/api/src/auth/auth.service.ts:119-122.

Correcao recomendada: documentar emissao/rotacao/revogacao de bearer tokens, reduzir TTL quando possivel, amarrar a device/session e auditar uso por IP/dispositivo.

MedioSEC-16 - Falha de billing libera acesso por seguranca operacional

No calculo de billing, excecoes retornam ACTIVE. Isso favorece disponibilidade, mas pode liberar tenants suspensos/expirados durante falhas de banco/consulta.

Evidencia: apps/api/src/trpc/trpc.controller.ts, funcao computeBillingState.

Correcao recomendada: definir fail-open apenas para falhas transientes conhecidas e registrar alerta; para status de suspensao, usar cache assinado/local com ultima decisao e expirar rapidamente.
